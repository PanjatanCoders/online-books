1. Introduction to Packages

Packages in Java provide a way to organize related classes and interfaces into a hierarchical structure. They serve as namespaces that help avoid naming conflicts and provide access control mechanisms.

Why Use Packages?

  • Organization: Group related classes logically
  • Namespace: Avoid naming conflicts between classes
  • Access Control: Control visibility of classes and members
  • Modularity: Enable better code maintenance and reusability
  • Security: Restrict access to sensitive code components

Real-World Analogy

Think of packages like folders on your computer. Just as you organize files into folders and subfolders, packages organize Java classes into a hierarchical directory structure.

2. Package Structure & Naming

Java packages follow a hierarchical naming convention that typically mirrors the directory structure of your project.

Standard Naming Convention

Package Naming Examples
// Company domain in reverse + project + module
package com.company.projectname.module;

// Examples:
package com.techtok.ecommerce.payment;
package com.techtok.ecommerce.inventory;
package com.techtok.ecommerce.user.authentication;

// Open source projects often use:
package org.apache.commons.lang;
package org.springframework.boot.autoconfigure;

Directory Structure Example

Project Directory Structure
src/
└── main/
    └── java/
        └── com/
            └── techtok/
                └── ecommerce/
                    ├── model/
                    │   ├── User.java
                    │   └── Product.java
                    ├── service/
                    │   ├── UserService.java
                    │   └── PaymentService.java
                    └── controller/
                        ├── UserController.java
                        └── ProductController.java

Naming Rules

  • Package names must be lowercase
  • Use dots (.) to separate package levels
  • Start with reversed domain name for uniqueness
  • Avoid Java keywords and reserved words

3. Creating and Using Packages

Creating packages involves declaring the package at the top of your Java file and organizing files in the corresponding directory structure.

Package Declaration

User.java
// Package declaration must be first non-comment line
package com.techtok.ecommerce.model;

public class User {
    private String username;
    private String email;
    
    // Constructor
    public User(String username, String email) {
        this.username = username;
        this.email = email;
    }
    
    // Getters and setters
    public String getUsername() { return username; }
    public String getEmail() { return email; }
}

Using Classes from Other Packages

UserService.java
package com.techtok.ecommerce.service;

// Import statement to use User class
import com.techtok.ecommerce.model.User;
import java.util.List;
import java.util.ArrayList;

public class UserService {
    private List<User> users = new ArrayList<>();
    
    public void addUser(String username, String email) {
        User user = new User(username, email);
        users.add(user);
    }
    
    public User findUserByUsername(String username) {
        return users.stream()
            .filter(user -> user.getUsername().equals(username))
            .findFirst()
            .orElse(null);
    }
}

4. Import Statements

Import statements allow you to use classes from other packages without fully qualifying their names every time.

Types of Import Statements

Import Examples
package com.techtok.ecommerce.controller;

// Single class import (preferred)
import com.techtok.ecommerce.model.User;
import com.techtok.ecommerce.service.UserService;

// Multiple classes from same package
import java.util.List;
import java.util.ArrayList;
import java.util.HashMap;

// Wildcard import (use sparingly)
import java.util.*;

// Static import for constants/methods
import static java.lang.Math.PI;
import static java.lang.Math.sqrt;

public class UserController {
    private UserService userService = new UserService();
    
    public void calculateDistance() {
        // Using static import
        double result = sqrt(16) * PI;
    }
}

Import Best Practices

  • Specific imports: Import specific classes rather than using wildcards
  • Organize imports: Group by package hierarchy
  • Remove unused: Clean up unused import statements
  • Static imports: Use for frequently used constants/utility methods

Import Conflicts

When two classes have the same name from different packages, you must use fully qualified names or import only one and fully qualify the other.

Handling Name Conflicts
package com.techtok.example;

// Import one Date class
import java.util.Date;

public class DateExample {
    public void demonstrateConflict() {
        // Using imported Date
        Date utilDate = new Date();
        
        // Using fully qualified name for SQL Date
        java.sql.Date sqlDate = new java.sql.Date(System.currentTimeMillis());
    }
}

5. Access Modifiers Overview

Access modifiers in Java control the visibility and accessibility of classes, methods, and variables. They are fundamental to encapsulation and security in object-oriented programming.

Four Access Levels

  • public: Accessible from anywhere
  • protected: Accessible within package and subclasses
  • default (package-private): Accessible within same package only
  • private: Accessible within same class only
