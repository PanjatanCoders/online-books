Q1: == vs .equals() in Java

Question

Difference between == and .equals() in Java. Give examples.

Key Differences

  • == operator: Compares reference (memory address) for objects, values for primitives
  • .equals() method: Compares content/value of objects
Examples
// Primitives - == compares values
int a = 10, b = 10;
System.out.println(a == b); // true

// Strings
String s1 = "Hello";
String s2 = "Hello";
String s3 = new String("Hello");

System.out.println(s1 == s2);        // true (same reference in String pool)
System.out.println(s1 == s3);        // false (different objects)
System.out.println(s1.equals(s3));   // true (same content)

Best Practice

Always use .equals() for object comparison. Override equals() in custom classes for proper comparison.

Q2: HashMap vs Hashtable vs ConcurrentHashMap

Question

Explain HashMap vs Hashtable vs ConcurrentHashMap.

Comparison Table
Feature              | HashMap         | Hashtable       | ConcurrentHashMap
---------------------+-----------------+-----------------+------------------
Thread Safety        | No              | Yes             | Yes
Null Keys/Values     | Allowed         | Not allowed     | Not allowed
Performance          | Fast            | Slow            | Fast
Synchronization      | Not synchronized| Synchronized    | Segment locking
Legacy               | No              | Yes             | No
Usage Examples
// HashMap - Not thread-safe
Map<String, Integer> hashMap = new HashMap<>();
hashMap.put("A", 1);
hashMap.put(null, 2); // Allows null

// Hashtable - Thread-safe but slow
Map<String, Integer> hashtable = new Hashtable<>();
hashtable.put("A", 1);
// hashtable.put(null, 2); // Throws NullPointerException

// ConcurrentHashMap - Thread-safe and fast
Map<String, Integer> concurrentMap = new ConcurrentHashMap<>();
concurrentMap.put("A", 1);
// concurrentMap.put(null, 2); // Throws NullPointerException

When to Use

HashMap for single-threaded, ConcurrentHashMap for multi-threaded applications. Avoid Hashtable (legacy).

Q3: ArrayList vs LinkedList

Question

What is the difference between ArrayList and LinkedList? When would you use which?

Key Differences

  • ArrayList: Dynamic array, fast random access O(1), slow insertion/deletion O(n)
  • LinkedList: Doubly-linked list, slow random access O(n), fast insertion/deletion O(1)
Usage
// ArrayList - Use for frequent access
List<String> arrayList = new ArrayList<>();
arrayList.add("A");
String element = arrayList.get(0); // O(1)

// LinkedList - Use for frequent add/remove
List<String> linkedList = new LinkedList<>();
linkedList.add(0, "A"); // O(1) at beginning
linkedList.remove(0);   // O(1) at beginning

Recommendation

Use ArrayList when you need frequent access by index. Use LinkedList when you need frequent insertions/deletions at beginning/middle.

Q4: final, finally, and finalize()

Question

Explain final, finally, and finalize() with examples.

Definitions

  • final: Keyword to make variables constant, methods non-overridable, classes non-inheritable
  • finally: Block that always executes after try-catch (used for cleanup)
  • finalize(): Method called by garbage collector before object destruction (deprecated)
Examples
// final keyword
final int MAX_VALUE = 100;
// MAX_VALUE = 200; // Compilation error

final class FinalClass { } // Cannot be extended

// finally block
try {
    // risky code
} catch (Exception e) {
    // handle exception
} finally {
    // Always executes - cleanup code
    System.out.println("Cleanup");
}

// finalize() - deprecated
@Override
protected void finalize() throws Throwable {
    // Called before GC (don't rely on this)
}

Q5: Palindrome Check in Java

Question

Write a Java program to check if a string is a palindrome.

Implementation
public class PalindromeChecker {

    // Method 1: Using two pointers
    public static boolean isPalindrome(String str) {
        str = str.toLowerCase().replaceAll("[^a-z0-9]", "");
        int left = 0, right = str.length() - 1;

        while (left < right) {
            if (str.charAt(left) != str.charAt(right)) {
                return false;
            }
            left++;
            right--;
        }
        return true;
    }

    // Method 2: Using StringBuilder
    public static boolean isPalindromeReverse(String str) {
        String cleaned = str.toLowerCase().replaceAll("[^a-z0-9]", "");
        String reversed = new StringBuilder(cleaned).reverse().toString();
        return cleaned.equals(reversed);
    }
}

Complexity

O(n) time, O(1) space for method 1. Method 2 uses O(n) extra space.

Q6: String Pool in Java

Question

Explain the concept of String Pool in Java.

String Pool is a special memory region in heap where Java stores string literals to optimize memory by reusing immutable strings.

String Pool Examples
// String literals go to pool
String s1 = "Hello";
String s2 = "Hello";
System.out.println(s1 == s2); // true (same reference)

// new String() creates object in heap
String s3 = new String("Hello");
System.out.println(s1 == s3); // false (different reference)

// intern() moves to pool
String s4 = s3.intern();
System.out.println(s1 == s4); // true (same reference)

Key Point

String pool saves memory by reusing string literals. Use intern() to move strings to pool.

Q7: Checked vs Unchecked Exceptions

Question

What are checked and unchecked exceptions? Give examples.

Exception Types

  • Checked Exceptions: Compile-time exceptions, must be handled or declared (IOException, SQLException)
  • Unchecked Exceptions: Runtime exceptions, optional to handle (NullPointerException, ArrayIndexOutOfBoundsException)
Examples
// Checked Exception - Must handle
public void readFile() throws IOException {
    FileReader file = new FileReader("test.txt");
}

// Or use try-catch
try {
    FileReader file = new FileReader("test.txt");
} catch (IOException e) {
    e.printStackTrace();
}

// Unchecked Exception - Optional to handle
int[] arr = {1, 2, 3};
System.out.println(arr[5]); // ArrayIndexOutOfBoundsException

String str = null;
System.out.println(str.length()); // NullPointerException

Q8: Static Keyword in Java

Question

Explain static keyword with different use cases.

Static Keyword Uses
public class StaticExample {

    // 1. Static variable - shared across all instances
    static int count = 0;

    // 2. Static method - can be called without object
    public static void displayCount() {
        System.out.println("Count: " + count);
    }

    // 3. Static block - executes once when class loads
    static {
        System.out.println("Static block executed");
        count = 10;
    }

    // 4. Static nested class
    static class NestedClass {
        void display() {
            System.out.println("Static nested class");
        }
    }
}

// Usage
StaticExample.displayCount(); // No object needed
StaticExample.count = 5;

Key Point

Static members belong to class, not instances. Useful for utility methods, constants, and counters.

Q9: throw vs throws

Question

Difference between throw and throws.

Key Differences

  • throw: Used to explicitly throw an exception from code
  • throws: Used in method signature to declare exceptions method might throw
Examples

    if (age < 18) {
        throw new IllegalArgumentException("Age must be 18+");
    }
}

// throws - declare exception in signature

    FileReader file = new FileReader(path);
}

// Can declare multiple exceptions

    // method body
}

Q10: Remove Duplicates from Array

Question

Write a Java program to remove duplicates from an array.

Implementation
// Method 1: Using HashSet
public static int[] removeDuplicates(int[] arr) {
    Set<Integer> set = new LinkedHashSet<>();
    for (int num : arr) {
        set.add(num);
    }
    return set.stream().mapToInt(Integer::intValue).toArray();
}

// Method 2: Using Stream API
public static int[] removeDuplicatesStream(int[] arr) {
    return Arrays.stream(arr).distinct().toArray();
}

// Usage
int[] arr = {1, 2, 2, 3, 4, 4, 5};
int[] result = removeDuplicates(arr);
// Output: {1, 2, 3, 4, 5}

Q11: findElement() vs findElements()

Question

Difference between findElement() and findElements().

Key Differences

  • findElement(): Returns single WebElement, throws NoSuchElementException if not found
  • findElements(): Returns List<WebElement>, returns empty list if none found

// findElements() - Returns list

System.out.println("Total elements: " + elements.size());

// Check if element exists
if (driver.findElements(By.id("optional")).size() > 0) {
    // Element exists
}

Q12: Handling Dynamic Web Elements

Question

How do you handle dynamic web elements in Selenium?

Strategies

  • Use Explicit Waits with Expected Conditions
  • Use dynamic XPath/CSS with contains(), starts-with()
  • Use relative locators
  • Handle stale element exceptions with retry logic
Implementation
// Wait for element to be clickable
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement element = wait.until(
    ExpectedConditions.elementToBeClickable(By.id("dynamic-btn")));

// Dynamic XPath
String xpath = "//button[contains(@id, 'submit')]";
driver.findElement(By.xpath(xpath)).click();

// Handle stale element
for (int i = 0; i < 3; i++) {
    try {
        driver.findElement(By.id("element")).click();
        break;
    } catch (StaleElementReferenceException e) {
        // Retry
    }
}

Q13: Types of Waits in Selenium

Question

Explain different types of waits in Selenium (implicit, explicit, fluent).

Wait Types
// 1. Implicit Wait - Global setting
driver.manage().timeouts().implicitlyWait(Duration.ofSeconds(10));

// 2. Explicit Wait - Specific conditions
WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
WebElement element = wait.until(
    ExpectedConditions.elementToBeClickable(By.id("submit")));

// 3. Fluent Wait - Custom polling and exceptions
FluentWait<WebDriver> fluentWait = new FluentWait<>(driver)
    .withTimeout(Duration.ofSeconds(30))
    .pollingEvery(Duration.ofSeconds(2))
    .ignoring(NoSuchElementException.class);

WebElement el = fluentWait.until(driver ->
    driver.findElement(By.id("dynamic-element")));

Important

Never mix implicit and explicit waits. Prefer explicit waits for better control.

XPath Solution

Q18: Relative vs Absolute XPath

Question

Difference between relative and absolute XPath. Which one do you prefer?

Key Differences

  • Absolute XPath: Starts from root (/) - Fragile, breaks with DOM changes
  • Relative XPath: Starts from anywhere (//) - Flexible, more maintainable
Examples
// Absolute XPath - Avoid
/html/body/div[1]/div[2]/form/input[1]

// Relative XPath - Preferred
//input[@id='username']
//form[@id='loginForm']//input[@name='username']

Recommendation

Always use relative XPath for better maintainability and resilience to DOM changes.

String mainWindow = driver.getWindowHandle();

// Get all window handles
Set<String> allWindows = driver.getWindowHandles();

// Switch to new window
for (String window : allWindows) {
    if (!window.equals(mainWindow)) {

    }

        switch(browser.toLowerCase()) {

Q28: Retry Failed Tests

Question

Explain how you handle retries for failed test cases.

Retry Analyzer
public class RetryAnalyzer implements IRetryAnalyzer {
    private int count = 0;
    private int maxRetry = 2;

    @Override
    public boolean retry(ITestResult result) {
        if (count < maxRetry) {
            count++;
            return true;
        }
        return false;
    }
}

// Use in test
@Test(retryAnalyzer = RetryAnalyzer.class)
public void testMethod() {
    // Test logic
}

Q30: Framework Maintainability & Scalability

Question

How do you ensure maintainability and scalability of your framework?

Maintainability

  • Page Object Model: Separate page logic from tests
  • Modular Design: Small, reusable methods
  • Clear Naming: Self-documenting code
  • Version Control: Git with proper branching

Scalability

  • Configuration Files: Externalize test data and settings
  • Parallel Execution: Run tests concurrently
  • Selenium Grid: Distribute tests across machines
  • Cloud Platforms: BrowserStack, Sauce Labs
Framework Architecture
Key Design Principles:

1. Separation of Concerns
   - Tests separate from page objects
   - Utils separate from business logic

2. DRY (Don't Repeat Yourself)
   - Reusable methods in base classes
   - Common utilities centralized

3. Configuration Management
   - Properties files for environments
   - Easy switching between DEV/QA/PROD

4. Robust Error Handling
   - Try-catch blocks with meaningful messages
   - Retry logic for flaky tests

5. Comprehensive Reporting
   - Detailed logs and screenshots
   - Executive summary reports

Golden Rule

If you need to change the same code in multiple places, it needs refactoring. Design for change from day one.
