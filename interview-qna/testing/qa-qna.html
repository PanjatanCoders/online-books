Q3. What is Software Testing?

Definition

Software testing is the process of evaluating and verifying that a software application functions correctly, meets specified requirements, and is free of defects.

Key Points:

  • Purpose: Find bugs, ensure quality, validate requirements
  • Types: Manual testing (human execution) vs Automated testing (tool-based)
  • Goal: Deliver reliable, high-quality software to end users
Interview Tip

Always emphasize that testing is not just about finding bugs - it's about ensuring the software meets user expectations and business requirements.

Q4. What are the different levels of testing?
1
Unit Testing
2
Integration Testing
3
System Testing
4
Acceptance Testing

1. Unit Testing

  • Test individual components/modules in isolation
  • Usually done by developers
  • Fastest and cheapest to fix defects

2. Integration Testing

  • Test interaction between integrated modules
  • Types: Big Bang, Top-down, Bottom-up, Sandwich

3. System Testing

  • Test complete integrated system
  • Verify system meets specified requirements
  • End-to-end functionality testing

4. Acceptance Testing

  • Final testing before release
  • Types: User Acceptance Testing (UAT), Business Acceptance Testing (BAT)

Q9. What is Regression Testing?

Definition

Testing existing functionality after changes to ensure new code doesn't break working features.

When to perform:

  • After bug fixes
  • After new feature additions
  • After code refactoring
  • Before releases
Full Regression

Test entire application

Partial Regression

Test affected areas only

Unit Regression

Test individual units

Progressive Regression

Test new functionality

Q10. What is Smoke and Sanity Testing?
Smoke Testing Sanity Testing
Wide and Shallow Narrow and Deep
Tests basic functionality Tests specific functionality after changes
Build acceptance testing Subset of regression testing
"Can we proceed with detailed testing?" "Is the specific fix working correctly?"
Banking App Example:

Smoke: Login works, main menu loads, basic navigation functional

Sanity: After fixing transfer bug, verify transfer functionality works correctly

Q11. What is Exploratory Testing?

Definition

Simultaneous learning, test design, and test execution without predetermined test cases.

Characteristics:

  • Investigative approach
  • Real-time decision making
  • Unscripted testing
  • Focus on discovery and learning

Benefits:

  • Finds unexpected issues
  • Improves test coverage
  • Provides quick feedback
  • Complements scripted testing
When to Use Exploratory Testing

Best used when requirements are unclear, time is limited, or when looking for usability issues that scripted tests might miss.

Q12. How do you write a good test case?

Good Test Case Characteristics:

  • Clear and Concise: Easy to understand
  • Traceable: Linked to requirements
  • Repeatable: Consistent results
  • Independent: Not dependent on other test cases
Test Case ID: TC_001 Test Scenario: User Login Pre-conditions: User account exists Test Steps: 1. Navigate to login page 2. Enter valid username 3. Enter valid password 4. Click login button Expected Result: User redirected to dashboard Post-conditions: User logged in successfully Priority: High Test Data: Username: testuser@email.com, Password: Test123!
Best Practices
  • Use action verbs (Click, Enter, Verify)
  • Include specific test data
  • Write from user perspective
  • Keep steps simple and atomic

Q14. What is a Use Case? How do you test it?

Use Case Definition

Description of how users interact with system to achieve specific goals.

Testing Approach:

  1. Identify Actors: Who uses the system?
  2. Map Scenarios: Normal flow, alternative flows, exception flows
  3. Create Test Cases: Cover all flows
  4. Validate: Ensure user goals are met
Example: Online Purchase

Actor: Customer

Normal Flow: Browse ‚Üí Select ‚Üí Add to Cart ‚Üí Checkout ‚Üí Payment ‚Üí Confirmation

Stage Descriptions:

  • New: Defect reported by tester
  • Assigned: Assigned to developer
  • Open: Developer working on fix

  • Verified: Fix confirmed working

Q20. How do you handle a situation where a developer disagrees with your bug?
1
Document Evidence
2
Reference Requirements
3
Facilitate Discussion
4
Seek Resolution

Professional Approach:

  • Document Evidence: Provide clear reproduction steps, screenshots
  • Reference Requirements: Show how behavior violates specifications
  • Facilitate Discussion: Arrange meeting with BA/PM if needed
  • Seek Second Opinion: Ask team lead or senior tester
  • Follow Process: Use established escalation procedures
  • Maintain Professionalism: Focus on facts, not personal opinions
Remember

The goal is to deliver quality software, not to be "right". Stay objective and collaborative.

Q22. What is a user story?

Format

"As a [user type], I want [functionality] so that [benefit]"

Example: As a bank customer, I want to check my account balance online So that I can monitor my finances without visiting the branch

Components:

  • Role: Who wants the feature
  • Goal: What they want to do
  • Benefit: Why they want it
Testing Perspective

User stories help testers understand the user's perspective and create more meaningful test scenarios.

Q23. What is acceptance criteria?

Definition

Conditions that must be met for user story to be considered complete.

User Story: Account balance check Acceptance Criteria: ‚úì User must be logged in ‚úì Balance displays current amount ‚úì Balance updates reflect recent transactions ‚úì Error message for network issues ‚úì Response time under 3 seconds
Testing Value

Acceptance criteria directly translate to test cases and help define "Definition of Done".

Q26. What challenges have you faced in testing finance-related modules?

Common Challenges:

  1. Data Sensitivity: Cannot use production data for testing
  2. Complex Business Rules: Multiple validation scenarios
  3. Regulatory Compliance: Strict adherence to financial regulations
  4. Integration Complexity: Multiple system dependencies
  5. Security Requirements: Encryption, PCI DSS compliance
  6. Performance: High transaction volumes
  7. Currency/Regional Variations: Different formats and rules
Critical Considerations

Financial applications require extra attention to data accuracy, security vulnerabilities, and regulatory compliance. Any bug in production can have serious financial and legal consequences.

Common Vulnerabilities to Test:

  • SQL Injection: Malicious SQL queries
  • Cross-site Scripting (XSS): Malicious scripts
  • Session Hijacking: Unauthorized session access
  • CSRF Attacks: Cross-site request forgery

Q29. What will you do if the requirements are unclear?
1
Document Questions
2
Reach Out
3
Schedule Meeting
4
Document & Confirm

Action Plan:

  1. Document Questions: List all unclear points systematically
  2. Reach Out: Contact BA, PM, or stakeholders
  3. Schedule Meeting: Arrange face-to-face or virtual discussion
  4. Document Assumptions: If clarification is unavailable
  5. Get Written Confirmation: Email confirmations of discussions
  6. Update Test Cases: Modify based on clarifications
Pro Tip

Never assume! It's better to ask questions and look thorough than to test the wrong functionality.

Q30. How do you test when you have limited time?

Risk-Based Testing Approach:

  1. Prioritize: Focus on critical/high-risk functionality first
  2. Smoke Testing: Cover basic functionality quickly
  3. Automate: Use automation for repetitive tasks
  4. Parallel Testing: Divide work among team members
  5. Exploratory Testing: Efficient for finding quick issues
  6. Communicate Risks: Inform stakeholders of coverage limitations
Time-boxed Strategy Example:

2 hours available:

  • 30 min: Smoke testing (basic flows)
  • 60 min: Critical business scenarios
  • 30 min: Exploratory testing on new features
Q31. A bug is found in production. How do you handle it?
1
Assess Impact
2
Document & Reproduce
3
Escalate
4
Root Cause Analysis

Immediate Actions:

  1. Assess Impact: Critical/blocking vs minor issue
  2. Document: Create detailed bug report
  3. Reproduce: Verify in production environment
  4. Escalate: Notify management for critical issues
  5. Hotfix Testing: Quick verification of fix
  6. Root Cause Analysis: Why did it reach production?
  7. Process Improvement: Strengthen testing processes
Production Bug Response

Speed and accuracy are crucial. Document everything, communicate clearly, and focus on resolution first, analysis second.

Q32. How do you test a login page?

Functional Testing:

  • Valid credentials ‚Üí Successful login
  • Invalid username ‚Üí Error message
  • Invalid password ‚Üí Error message
  • Empty fields ‚Üí Validation messages
  • Password visibility toggle
  • Remember me functionality
  • Forgot password link

Non-Functional Testing:

  • Security: SQL injection, XSS attempts
  • Performance: Response time under load
  • Usability: User-friendly error messages
  • Compatibility: Different browsers/devices
Test Cases Example: TC_001: Valid login Steps: Enter valid username/password, click login Expected: Redirect to dashboard TC_002: Invalid password Steps: Enter valid username, invalid password, click login Expected: Error message "Invalid password" TC_003: Account lockout Steps: Enter wrong password 3 times Expected: Account locked for security TC_004: SQL Injection attempt Steps: Enter "admin'; DROP TABLE users;--" as username Expected: Input sanitized, login fails safely
