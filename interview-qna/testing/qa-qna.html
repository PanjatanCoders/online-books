Testing Fundamentals

Core concepts every QA professional must know for interviews.

Q1: What is Software Testing?

Software testing is the process of evaluating and verifying that a software application functions correctly, meets specified requirements, and is free of defects.

Key Points

  • Purpose: Find bugs, ensure quality, validate requirements
  • Types: Manual testing (human execution) vs Automated testing (tool-based)
  • Goal: Deliver reliable, high-quality software to end users

Interview Tip

Always emphasize that testing is not just about finding bugs - it's about ensuring the software meets user expectations and business requirements.

Q2: What are the different levels of testing?

Testing Levels
1. Unit Testing
   - Test individual components/modules in isolation
   - Usually done by developers
   - Fastest and cheapest to fix defects

2. Integration Testing
   - Test interaction between integrated modules
   - Types: Big Bang, Top-down, Bottom-up, Sandwich

3. System Testing
   - Test complete integrated system
   - Verify system meets specified requirements
   - End-to-end functionality testing

4. Acceptance Testing
   - Final testing before release
   - Types: User Acceptance Testing (UAT), Business Acceptance Testing (BAT)

Q3: Functional vs Non-functional Testing

Comparison
Functional Testing          | Non-functional Testing
----------------------------+---------------------------
Tests WHAT the system does  | Tests HOW the system performs
Validates business requirements | Validates system characteristics
Input → Process → Output    | Performance, security, usability
Examples: Login, payments   | Examples: Load testing, security

Q4: Verification vs Validation

Verification

  • "Are we building the product right?"
  • Static testing (no code execution)
  • Reviews, inspections, walkthroughs
  • Done during development

Validation

  • "Are we building the right product?"
  • Dynamic testing (code execution)
  • Actual testing with test cases
  • Done after development

Q5: Software Testing Life Cycle (STLC)

STLC Phases
1. Requirement Analysis
   └─ Understand and analyze requirements, identify testable requirements

2. Test Planning
   └─ Create test strategy and plan, estimate effort and resources

3. Test Case Development
   └─ Write detailed test cases, prepare test data

4. Environment Setup
   └─ Configure test environment, set up test data

5. Test Execution
   └─ Execute test cases, log defects

6. Test Closure
   └─ Test completion report, lessons learned

Q7: What is Regression Testing?

Testing existing functionality after changes to ensure new code doesn't break working features.

When to Perform

  • After bug fixes
  • After new feature additions
  • After code refactoring
  • Before releases

Full Regression

Test entire application

Partial Regression

Test affected areas only

Q8: Smoke vs Sanity Testing

Comparison
Smoke Testing                     | Sanity Testing
----------------------------------+----------------------------------
Wide and Shallow                  | Narrow and Deep
Tests basic functionality         | Tests specific functionality
Build acceptance testing          | Subset of regression testing
"Can we proceed with testing?"    | "Is the specific fix working?"

Banking App Example

  • Smoke: Login works, main menu loads, basic navigation functional
  • Sanity: After fixing transfer bug, verify transfer functionality works correctly

Q9: What is Exploratory Testing?

Simultaneous learning, test design, and test execution without predetermined test cases.

Characteristics

  • Investigative approach
  • Real-time decision making
  • Unscripted testing
  • Focus on discovery

Benefits

  • Finds unexpected issues
  • Improves test coverage
  • Provides quick feedback
  • Complements scripted testing

When to Use

Best used when requirements are unclear, time is limited, or when looking for usability issues that scripted tests might miss.

Test Design & Execution

Creating effective test cases and ensuring thorough test coverage.

Q10: How to Write a Good Test Case?

Good Test Case Characteristics

  • Clear and Concise: Easy to understand
  • Traceable: Linked to requirements
  • Repeatable: Consistent results
  • Independent: Not dependent on other test cases
Test Case Template
Test Case ID: TC_001
Test Scenario: User Login
Pre-conditions: User account exists

Test Steps:
1. Navigate to login page
2. Enter valid username
3. Enter valid password
4. Click login button

Expected Result: User redirected to dashboard
Post-conditions: User logged in successfully
Priority: High
Test Data: Username: testuser@email.com, Password: Test123!

Best Practices

  • Use action verbs (Click, Enter, Verify)
  • Include specific test data
  • Write from user perspective
  • Keep steps simple and atomic

Q11: How to Ensure Complete Test Coverage?

Requirements Traceability Matrix (RTM)

  • Map test cases to requirements
  • Ensure all requirements tested

Test Coverage Techniques

  • Boundary Value Analysis
  • Equivalence Partitioning
  • Decision Table Testing
  • Peer reviews of test cases
  • Risk-based testing

Q12: How to Prioritize Test Cases?

Priority Factors

  1. Business Impact: Critical functionality first
  2. Risk Assessment: High-risk areas priority
  3. Customer Usage: Frequently used features
  4. Complexity: Complex modules need more testing
Priority Levels
P0 - Critical: Blocker issues that prevent core functionality
P1 - High:     Major functionality that impacts business
P2 - Medium:   Important features with moderate impact
P3 - Low:      Nice to have features

Defect Management

Bug lifecycle, reporting, and handling disagreements.

Q13: What is the Defect Life Cycle?

Defect States
New → Assigned → Open → Fixed → Retest → Verified → Closed

Alternative paths:
- New → Rejected (not a valid defect)
- Fixed → Reopened (issue persists)
- Open → Deferred (fix postponed)

Stage Descriptions:
- New:      Defect reported by tester
- Assigned: Assigned to developer
- Open:     Developer working on fix
- Fixed:    Developer completed fix
- Retest:   Ready for testing
- Verified: Fix confirmed working
- Closed:   Defect resolved

Q14: What to Include When Raising a Bug?

Bug Report Template
Bug ID: BUG_001
Summary: Login fails with valid credentials
Description: User cannot login with correct username/password
Severity: High | Priority: High
Environment: Windows 10, Chrome 96.0

Steps to Reproduce:
1. Navigate to login page
2. Enter username: test@email.com
3. Enter password: Test123!
4. Click Login button

Expected Result: User should be logged in
Actual Result: Error message "Invalid credentials"
Screenshots: [attachment]
Reporter: John Doe
Date: 2024-01-15

Essential Components

  • Bug ID: Unique identifier
  • Summary: Brief description
  • Steps to Reproduce: Detailed steps
  • Severity/Priority: Impact assessment
  • Environment: OS, browser, version
  • Expected vs Actual Result: Clear comparison
  • Screenshots: Visual evidence

Q15: Severity vs Priority

Comparison
Severity (Impact on functionality) | Priority (Business urgency)
-----------------------------------+----------------------------
Technical assessment               | Business decision
High: System crash                 | High: Must fix immediately
Medium: Feature not working        | Medium: Fix in current release
Low: UI cosmetic issues            | Low: Fix in future release

Examples

  • High Severity, Low Priority: Crash in rarely used admin feature
  • Low Severity, High Priority: Company logo missing on homepage

Q16: Handling Developer Disagreement

Question

How do you handle a situation where a developer disagrees with your bug?

Professional Approach

  1. Document Evidence: Provide clear reproduction steps, screenshots
  2. Reference Requirements: Show how behavior violates specifications
  3. Facilitate Discussion: Arrange meeting with BA/PM if needed
  4. Seek Second Opinion: Ask team lead or senior tester
  5. Follow Process: Use established escalation procedures
  6. Maintain Professionalism: Focus on facts, not personal opinions

Remember

The goal is to deliver quality software, not to be "right". Stay objective and collaborative.

Q18: What is a User Story?

User Story Format

"As a [user type], I want [functionality] so that [benefit]"

Example
As a bank customer,
I want to check my account balance online
So that I can monitor my finances without visiting the branch

Components

  • Role: Who wants the feature
  • Goal: What they want to do
  • Benefit: Why they want it

Q19: What is Acceptance Criteria?

Conditions that must be met for user story to be considered complete.

Example
User Story: Account balance check

Acceptance Criteria:
- User must be logged in
- Balance displays current amount
- Balance updates reflect recent transactions
- Error message for network issues
- Response time under 3 seconds

Testing Value

Acceptance criteria directly translate to test cases and help define "Definition of Done".

Q22: Challenges in Finance Module Testing

Common Challenges

  1. Data Sensitivity: Cannot use production data for testing
  2. Complex Business Rules: Multiple validation scenarios
  3. Regulatory Compliance: Strict adherence to financial regulations
  4. Integration Complexity: Multiple system dependencies
  5. Security Requirements: Encryption, PCI DSS compliance
  6. Performance: High transaction volumes
  7. Currency/Regional Variations: Different formats and rules

Critical Note

Financial applications require extra attention to data accuracy, security vulnerabilities, and regulatory compliance. Any bug in production can have serious financial and legal consequences.

Q23: Security Testing in Banking Apps

Authentication Testing

Login security, password policies, MFA

Authorization Testing

Role-based access, transaction limits

Data Protection

Encryption, PII protection

Session Management

Timeout, session hijacking prevention

Common Vulnerabilities to Test

  • SQL Injection: Malicious SQL queries
  • Cross-site Scripting (XSS): Malicious scripts
  • Session Hijacking: Unauthorized session access
  • CSRF Attacks: Cross-site request forgery
  • Man-in-the-Middle: Data interception

Scenario-Based Questions

Real-world situations and how to handle them.

Q24: Handling Unclear Requirements

Action Plan

  1. Document Questions: List all unclear points systematically
  2. Reach Out: Contact BA, PM, or stakeholders
  3. Schedule Meeting: Arrange face-to-face or virtual discussion
  4. Document Assumptions: If clarification is unavailable
  5. Get Written Confirmation: Email confirmations of discussions
  6. Update Test Cases: Modify based on clarifications

Pro Tip

Never assume! It's better to ask questions and look thorough than to test the wrong functionality.

Q25: Testing with Limited Time

Risk-Based Testing Approach

  1. Prioritize: Focus on critical/high-risk functionality first
  2. Smoke Testing: Cover basic functionality quickly
  3. Automate: Use automation for repetitive tasks
  4. Parallel Testing: Divide work among team members
  5. Exploratory Testing: Efficient for finding quick issues
  6. Communicate Risks: Inform stakeholders of coverage limitations
Time-boxed Strategy (2 hours)
30 min: Smoke testing (basic flows)
60 min: Critical business scenarios
30 min: Exploratory testing on new features

Q26: Handling Production Bugs

Immediate Actions

  1. Assess Impact: Critical/blocking vs minor issue
  2. Document: Create detailed bug report
  3. Reproduce: Verify in production environment
  4. Escalate: Notify management for critical issues
  5. Hotfix Testing: Quick verification of fix
  6. Root Cause Analysis: Why did it reach production?
  7. Process Improvement: Strengthen testing processes

Production Bug Response

Speed and accuracy are crucial. Document everything, communicate clearly, and focus on resolution first, analysis second.

Q27: How to Test a Login Page?

Functional Testing

  • Valid credentials - Successful login
  • Invalid username - Error message
  • Invalid password - Error message
  • Empty fields - Validation messages
  • Password visibility toggle
  • Remember me functionality
  • Forgot password link

Non-Functional Testing

  • Security: SQL injection, XSS attempts
  • Performance: Response time under load
  • Usability: User-friendly error messages
  • Compatibility: Different browsers/devices
Sample Test Cases
TC_001: Valid login
Steps: Enter valid username/password, click login
Expected: Redirect to dashboard

TC_002: Invalid password
Steps: Enter valid username, invalid password, click login
Expected: Error message "Invalid password"

TC_003: Account lockout
Steps: Enter wrong password 3 times
Expected: Account locked for security

TC_004: SQL Injection attempt
Steps: Enter "admin'; DROP TABLE users;--" as username
Expected: Input sanitized, login fails safely

Interview Success Tips

Best Practices

  • Always provide specific examples from your experience
  • Demonstrate knowledge of testing tools (JIRA, TestRail, Selenium, Postman)
  • Show understanding of both manual and automation testing
  • Emphasize collaboration and communication skills
  • Prepare questions about their testing processes and tools
  • Practice explaining concepts in simple terms
  • Be ready with real scenarios from your testing experience

Common Mistakes to Avoid

  • Not asking clarifying questions - Always seek to understand the question fully
  • Being too technical or too basic - Match your answer to the interviewer's level
  • Not providing real examples - Always back up your knowledge with experience
  • Speaking negatively about past employers - Focus on positive learnings
  • Not preparing questions - Show genuine interest in their testing practices

Questions to Ask Your Interviewer

  • What testing tools and frameworks does your team currently use?
  • How do you handle test data management and environment setup?
  • What's the ratio of manual to automated testing in your projects?
  • How does your team collaborate with developers during the development cycle?
  • What are the biggest testing challenges the team is currently facing?

Final Preparation Checklist

  • Review testing fundamentals - STLC, test levels, types
  • Practice test case writing - Clear, concise, traceable
  • Understand defect lifecycle - States, severity vs priority
  • Know Agile methodology - User stories, acceptance criteria
  • Prepare domain examples - Banking, e-commerce, healthcare
  • Practice scenario-based answers - STAR method
