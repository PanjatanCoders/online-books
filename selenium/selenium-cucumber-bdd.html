Master Behavior-Driven Development with Cucumber and Selenium WebDriver for readable, maintainable test automation.

Introduction to BDD

What is Behavior-Driven Development?

BDD bridges the gap between business stakeholders and technical teams by using a common language to describe application behavior. Cucumber implements BDD using Gherkin syntax.

Why BDD?

  • Living Documentation - Tests serve as up-to-date specifications
  • Collaboration - Business, QA, and Dev speak the same language
  • Clarity - Plain English scenarios everyone understands
  • Reusability - Step definitions can be shared across scenarios

Gherkin Syntax

Feature File Basics

Feature files use .feature extension and contain scenarios written in Gherkin:

Feature: User Login
  As a registered user
  I want to login to my account
  So that I can access my dashboard

  @smoke @login
  Scenario: Successful login with valid credentials
    Given I am on the login page
    When I enter username "testuser"
    And I enter password "password123"
    And I click the login button
    Then I should see the dashboard
    And the welcome message should contain "testuser"

Gherkin Keywords

Keywords Reference

Keyword Purpose
FeatureDescribes the feature being tested
ScenarioA single test case
GivenPreconditions (setup)
WhenActions performed
ThenExpected outcomes
And/ButAdditional steps
BackgroundCommon steps for all scenarios
@tagTags for filtering/grouping

Background - Common Setup

Use Background for steps that repeat in every scenario:

Feature: Shopping Cart

  Background:
    Given I am logged in as a customer
    And I have an empty cart

  Scenario: Add single item to cart
    When I add "Laptop" to the cart
    Then the cart should contain 1 item

  Scenario: Add multiple items to cart
    When I add "Laptop" to the cart
    And I add "Mouse" to the cart
    Then the cart should contain 2 items

Step Definitions

Basic Step Definition

Step definitions map Gherkin steps to Java code:

public class LoginSteps {
    private WebDriver driver;
    private LoginPage loginPage;

    @Given("I am on the login page")
    public void iAmOnTheLoginPage() {
        driver = DriverManager.getDriver();
        driver.get("https://example.com/login");
        loginPage = new LoginPage(driver);
    }

    @When("I enter username {string}")
    public void iEnterUsername(String username) {
        loginPage.enterUsername(username);
    }

    @When("I enter password {string}")
    public void iEnterPassword(String password) {
        loginPage.enterPassword(password);
    }

    @When("I click the login button")
    public void iClickTheLoginButton() {
        loginPage.clickLogin();
    }

    @Then("I should see the dashboard")
    public void iShouldSeeTheDashboard() {
        Assert.assertTrue(loginPage.isDashboardDisplayed());
    }
}

Cucumber Expressions

Cucumber 7+ uses Cucumber Expressions (simpler than regex):

// String parameter
@When("I search for {string}")
public void searchFor(String query) { }

// Integer parameter
@Then("I should see {int} results")
public void verifyResults(int count) { }

// Word (no quotes needed)
@Given("I am on the {word} page")
public void navigateToPage(String pageName) { }

// Optional text
@When("I click (the )login button")
public void clickLogin() { }

// Alternative text
@When("I add/remove the item")
public void modifyCart() { }

Custom Parameter Types

@ParameterType("enabled|disabled")
public Boolean status(String status) {
    return "enabled".equals(status);
}

// Usage in step
@Then("the feature should be {status}")
public void verifyFeatureStatus(Boolean isEnabled) {
    Assert.assertEquals(isEnabled, feature.isEnabled());
}

Hooks

Setup and Teardown

Hooks run before/after scenarios for setup and cleanup:

public class Hooks {

    @Before
    public void setUp(Scenario scenario) {
        System.out.println("Starting: " + scenario.getName());
        DriverManager.initDriver();
    }

    @After
    public void tearDown(Scenario scenario) {
        // Capture screenshot on failure
        if (scenario.isFailed()) {
            byte[] screenshot = ((TakesScreenshot) DriverManager.getDriver())
                .getScreenshotAs(OutputType.BYTES);
            scenario.attach(screenshot, "image/png", "failure-screenshot");
        }
        DriverManager.quitDriver();
    }
}

Hook Types and Order

@BeforeAll   // Runs once before all scenarios
public static void globalSetup() {
    // Database setup, test data preparation
}

@AfterAll    // Runs once after all scenarios
public static void globalTeardown() {
    // Cleanup resources
}

@Before(order = 1)  // Lower order runs first
public void firstSetup() { }

@Before(order = 2)
public void secondSetup() { }

@Before("@database")  // Conditional hook - only for tagged scenarios
public void setupDatabase() { }

@BeforeStep
public void beforeEachStep() { }

@AfterStep
public void afterEachStep(Scenario scenario) {
    // Useful for detailed logging
}

Hook Order

Execution order: @BeforeAll → @Before → @BeforeStep → Step → @AfterStep → @After → @AfterAll

Data Tables

Simple Data Table

Scenario: Fill registration form
  When I fill in the registration form:
    | field     | value              |
    | firstName | John               |
    | lastName  | Doe                |
    | email     | john@example.com   |
    | phone     | 1234567890         |
  Then the form should be submitted successfully
@When("I fill in the registration form:")
public void fillRegistrationForm(DataTable dataTable) {
    Map<String, String> data = dataTable.asMap(String.class, String.class);

    registrationPage.enterFirstName(data.get("firstName"));
    registrationPage.enterLastName(data.get("lastName"));
    registrationPage.enterEmail(data.get("email"));
    registrationPage.enterPhone(data.get("phone"));
}

List of Maps (Multiple Records)

Scenario: Add multiple users
  When I create the following users:
    | username | email              | role    |
    | john     | john@test.com      | admin   |
    | jane     | jane@test.com      | user    |
    | bob      | bob@test.com       | viewer  |
@When("I create the following users:")
public void createUsers(DataTable dataTable) {
    List<Map<String, String>> users = dataTable.asMaps();

    for (Map<String, String> user : users) {
        userService.createUser(
            user.get("username"),
            user.get("email"),
            user.get("role")
        );
    }
}

TestNG Runner

@CucumberOptions(
    features = "src/test/resources/features",
    glue = {"stepdefinitions", "hooks"},
    plugin = {
        "pretty",
        "html:target/cucumber-reports/cucumber.html",
        "json:target/cucumber-reports/cucumber.json",
        "junit:target/cucumber-reports/cucumber.xml",
        "rerun:target/failed-scenarios.txt"
    },
    tags = "@smoke and not @skip",
    monochrome = true,
    dryRun = false
)
public class TestRunner extends AbstractTestNGCucumberTests {

    @Override
    @DataProvider(parallel = true)
    public Object[][] scenarios() {
        return super.scenarios();
    }
}

Runner Options Explained

CucumberOptions

Option Description
featuresPath to feature files
gluePackages containing step definitions
pluginReport formatters
tagsFilter scenarios by tags
dryRunCheck step mappings without execution
monochromeClean console output

Tag Expressions

// Single tag
tags = "@smoke"

// Multiple tags (AND)
tags = "@smoke and @login"

// Either tag (OR)
tags = "@smoke or @regression"

// Exclude tag
tags = "not @wip"

// Complex expression
tags = "(@smoke or @regression) and not @skip"

Running from Command Line

# Run all tests
mvn test

# Run specific tags
mvn test -Dcucumber.filter.tags="@smoke"

# Run with specific feature
mvn test -Dcucumber.features="src/test/resources/features/login.feature"

# Rerun failed scenarios
mvn test -Dcucumber.features="@target/failed-scenarios.txt"

Page Object Integration

Combining POM with Cucumber

Keep step definitions thin by delegating to Page Objects:

// Page Object
public class LoginPage extends BasePage {
    @FindBy(id = "username") private WebElement usernameField;
    @FindBy(id = "password") private WebElement passwordField;
    @FindBy(id = "login-btn") private WebElement loginButton;
    @FindBy(css = ".error-message") private WebElement errorMessage;

    public LoginPage(WebDriver driver) {
        super(driver);
    }

    public void login(String username, String password) {
        type(usernameField, username);
        type(passwordField, password);
        click(loginButton);
    }

    public String getErrorMessage() {
        return getText(errorMessage);
    }
}
// Step Definition - Clean and focused
public class LoginSteps {
    private LoginPage loginPage;
    private DashboardPage dashboardPage;

    @Given("I am on the login page")
    public void navigateToLogin() {
        DriverManager.getDriver().get(ConfigReader.getUrl("login"));
        loginPage = new LoginPage(DriverManager.getDriver());
    }

    @When("I login with {string} and {string}")
    public void performLogin(String username, String password) {
        loginPage.login(username, password);
        dashboardPage = new DashboardPage(DriverManager.getDriver());
    }

    @Then("I should see error {string}")
    public void verifyError(String expectedError) {
        Assert.assertEquals(expectedError, loginPage.getErrorMessage());
    }
}

Sharing State Between Steps

Using Dependency Injection

Share data between step definition classes using PicoContainer:

<dependency>
    <groupId>io.cucumber</groupId>
    <artifactId>cucumber-picocontainer</artifactId>
    <version>7.15.0</version>
</dependency>
// Shared context class
public class TestContext {
    private WebDriver driver;
    private Scenario scenario;
    private Map<String, Object> scenarioData = new HashMap<>();

    public WebDriver getDriver() {
        if (driver == null) {
            driver = DriverFactory.createDriver();
        }
        return driver;
    }

    public void setData(String key, Object value) {
        scenarioData.put(key, value);
    }

    @SuppressWarnings("unchecked")
    public <T> T getData(String key) {
        return (T) scenarioData.get(key);
    }

    public void quitDriver() {
        if (driver != null) {
            driver.quit();
            driver = null;
        }
    }
}
// Step definitions receive shared context via constructor
public class LoginSteps {
    private final TestContext context;
    private LoginPage loginPage;

    public LoginSteps(TestContext context) {
        this.context = context;
    }

    @When("I login as {string}")
    public void loginAs(String userType) {
        loginPage = new LoginPage(context.getDriver());
        User user = TestDataFactory.getUser(userType);
        loginPage.login(user.getUsername(), user.getPassword());
        context.setData("currentUser", user);
    }
}

public class DashboardSteps {
    private final TestContext context;

    public DashboardSteps(TestContext context) {
        this.context = context;
    }

    @Then("the welcome message should display my name")
    public void verifyWelcomeMessage() {
        User user = context.getData("currentUser");
        DashboardPage dashboard = new DashboardPage(context.getDriver());
        Assert.assertTrue(dashboard.getWelcomeMessage().contains(user.getName()));
    }
}

Extent Reports

// In Hooks class
private static ExtentReports extent;
private static ExtentTest feature;
private ExtentTest scenario;

@BeforeAll
public static void initReport() {
    extent = new ExtentReports();
    extent.attachReporter(new ExtentSparkReporter("reports/extent-report.html"));
}

@Before
public void beforeScenario(Scenario scenario) {
    this.scenario = extent.createTest(scenario.getName());
    scenario.getSourceTagNames().forEach(tag -> this.scenario.assignCategory(tag));
}

@AfterStep
public void afterStep(Scenario scenario) {
    if (scenario.isFailed()) {
        this.scenario.fail("Step failed");
    }
}

@AfterAll
public static void flushReport() {
    extent.flush();
}

Best Practices

Do

  • Write scenarios from user perspective
  • Keep scenarios independent
  • Use meaningful tag names
  • One assertion per Then step
  • Reuse step definitions
  • Keep step definitions thin
  • Use Background for common setup
  • Write declarative steps (what, not how)

Avoid

  • UI details in feature files
  • Scenarios longer than 10 steps
  • Coupling scenarios together
  • Technical jargon in Gherkin
  • Too many Scenario Outlines
  • Business logic in step definitions
  • Hard-coded test data
  • Imperative steps (click, type)

Good vs Bad Scenarios

Bad - Imperative Style

# Too detailed, hard to maintain
Scenario: Login
  Given I open browser
  And I navigate to "https://app.com/login"
  And I wait for page to load
  When I click on username field
  And I type "testuser"
  And I click on password field
  And I type "password123"
  And I click login button
  Then I should see element with id "dashboard"

Good - Declarative Style

# Business-focused, readable
Scenario: Successful login
  Given I am a registered user
  When I login with valid credentials
  Then I should see my dashboard
