Docker for QA Engineers

Run Selenium and Chrome inside Docker — no browser installation on your CI server, no driver version mismatches, tests that run identically everywhere.

10 min read Beginner Docker

The QA Pain Docker Eliminates

Before Docker, running Selenium tests on a CI server was painful:

  • Install Chrome on the Jenkins server — then it auto-updates and breaks your ChromeDriver version
  • Tests pass on your laptop (Chrome 120), fail on CI (Chrome 118)
  • Headless mode configuration differs between OS types
  • New team member spends a day setting up their test environment

Docker Fixes All of This

The Selenium team maintains official Docker images that include a pre-configured browser and Selenium server in one container. You pull it, run it, point your tests at it — done. Chrome version is pinned. Driver is already matched. Works on every OS.

The Selenium Docker Images

The Selenium project publishes ready-made images. The ones you will use most:

Image What is inside
selenium/standalone-chrome Chrome + ChromeDriver + Selenium Server, all-in-one
selenium/standalone-firefox Firefox + GeckoDriver + Selenium Server, all-in-one
selenium/standalone-edge Edge + EdgeDriver + Selenium Server, all-in-one

The "standalone" images are the easiest — everything you need in a single container. They also come with a VNC viewer so you can actually watch the browser run visually.

Run Chrome in One Command

Start a Selenium Chrome container:

docker run -d -p 4444:4444 --name selenium-chrome selenium/standalone-chrome:latest

That is it. You now have a fully functional Chrome browser running in Docker, with a Selenium Server listening on port 4444.

Flag Why it is there
-d Run in background so your terminal is free
-p 4444:4444 Expose the Selenium Server port to your machine
--name selenium-chrome Give it a name so you can stop it easily later

Verify it started. Open your browser and go to http://localhost:4444 — you will see the Selenium Grid UI showing one Chrome node ready.

Connect Your Tests

Instead of creating a ChromeDriver locally, you point your tests at the Selenium Server URL using RemoteWebDriver. 

ChromeOptions options = new ChromeOptions();

WebDriver driver = new RemoteWebDriver(
    new URL("http://localhost:4444"),
    options
);

That is three lines. Your tests now run in the Docker container.

Nothing Else Changes

All your existing test code — driver.get(), findElement(), click() — works exactly the same with RemoteWebDriver. You only changed how the driver was created.

Watch the Tests Run (Optional)

The selenium/standalone-chrome image comes with VNC — a remote desktop viewer. Add one more port when starting the container: 

docker run -d -p 4444:4444 -p 7900:7900 selenium/standalone-chrome:latest

Open your browser and go to http://localhost:7900. Password is secret. You will see a desktop with Chrome running your tests live.

Great for Debugging

When a test fails and you cannot see why, the VNC viewer shows you exactly what the browser is doing. Much easier than trying to interpret screenshots after the fact.

Pin the Browser Version

Using :latest is fine for quick experiments but risky for real CI pipelines. Pin to a specific version: 

docker run -d -p 4444:4444 selenium/standalone-chrome:4.18.1-20240224

Now your pipeline always uses the exact same Chrome version, no matter when it runs. Check the Selenium Docker Hub page for available version tags.

Clean Up After Tests

When your tests finish, stop and remove the container:

docker rm -f selenium-chrome

Or use the --rm flag when starting — it auto-removes the container once it stops: 

docker run -d -p 4444:4444 --rm selenium/standalone-chrome:latest

On CI Pipelines

In Jenkins or GitHub Actions, the typical pattern is: start the Selenium container before tests, run your tests, then stop it. Many CI systems support Docker natively and handle this automatically.
